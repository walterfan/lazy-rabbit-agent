## ADDED Requirements

### Requirement: Users can create learning goals

The system SHALL allow authenticated users to create learning goals with a subject, description, optional deadline, and optional daily study target.

#### Scenario: User creates a learning goal

- **WHEN** an authenticated user POSTs to `/api/v1/coach/goals` with `subject` and optional fields
- **THEN** the system creates a goal scoped to that user
- **AND** returns HTTP 201 with the goal ID, subject, status (`active`), and creation timestamp

#### Scenario: Goal creation requires subject

- **WHEN** the user POSTs without a `subject`
- **THEN** the system responds with 422 Unprocessable Entity

### Requirement: Users can list their learning goals

The system SHALL allow authenticated users to list their own learning goals, optionally filtered by status.

#### Scenario: User lists active goals

- **WHEN** an authenticated user GETs `/api/v1/coach/goals?status=active`
- **THEN** the system returns only active goals belonging to that user

#### Scenario: User lists all goals

- **WHEN** an authenticated user GETs `/api/v1/coach/goals` without status filter
- **THEN** the system returns all goals (active, completed, paused) belonging to that user

### Requirement: Users can update learning goal status

The system SHALL allow authenticated users to update their goal's status (active, paused, completed) and other fields.

#### Scenario: User completes a goal

- **WHEN** an authenticated user PATCHes `/api/v1/coach/goals/{id}` with `status=completed`
- **AND** the goal belongs to that user
- **THEN** the system updates the goal status and sets `completed_at` timestamp

#### Scenario: User cannot update another user's goal

- **WHEN** user A attempts to update a goal belonging to user B
- **THEN** the system responds with 404 Not Found

### Requirement: Users can log study sessions

The system SHALL allow authenticated users to log study sessions linked to a learning goal.

#### Scenario: User logs a study session

- **WHEN** an authenticated user POSTs to `/api/v1/coach/sessions` with `goal_id`, `duration_minutes`, and optional `notes` and `difficulty`
- **THEN** the system creates a study session record
- **AND** returns HTTP 201 with the session details

#### Scenario: Study session requires valid goal

- **WHEN** the user POSTs a session with a `goal_id` that doesn't exist or doesn't belong to them
- **THEN** the system responds with 404 Not Found

#### Scenario: Difficulty is validated

- **WHEN** the user provides `difficulty` outside the range 1-5
- **THEN** the system responds with 422 Unprocessable Entity

### Requirement: Users can list study sessions for a goal

The system SHALL allow authenticated users to list study sessions for a specific goal.

#### Scenario: User lists sessions for a goal

- **WHEN** an authenticated user GETs `/api/v1/coach/sessions?goal_id={id}`
- **THEN** the system returns all study sessions for that goal, ordered by date descending

### Requirement: Progress report provides comprehensive tracking data

The system SHALL provide a progress report endpoint that calculates and returns learning metrics for a specific goal.

#### Scenario: User gets progress report

- **WHEN** an authenticated user GETs `/api/v1/coach/progress/{goal_id}`
- **THEN** the system returns:
  - `total_sessions`: number of study sessions logged
  - `total_minutes`: total study time in minutes
  - `current_streak`: consecutive days with at least one session
  - `longest_streak`: longest consecutive day streak
  - `completion_percentage`: estimated completion (based on daily target vs actual)
  - `avg_difficulty`: average difficulty rating across sessions
  - `last_session_date`: date of most recent session
  - `ai_feedback`: AI-generated personalized feedback based on the progress data

#### Scenario: Progress report for goal with no sessions

- **WHEN** the user requests progress for a goal with no logged sessions
- **THEN** the system returns zeroed metrics and encouraging AI feedback to get started

#### Scenario: Streak calculation is accurate

- **WHEN** the user has logged sessions on 5 consecutive days, skipped 1 day, then logged 2 more days
- **THEN** `current_streak` is 2 and `longest_streak` is 5

### Requirement: AI feedback in progress report is personalized

The AI feedback in the progress report SHALL be generated by the LLM based on the user's actual progress data, goal details, and study patterns.

#### Scenario: AI feedback acknowledges good progress

- **WHEN** the user has maintained a streak and is on track with their daily target
- **THEN** the AI feedback includes encouragement and suggests next steps

#### Scenario: AI feedback addresses falling behind

- **WHEN** the user has missed several days or is below their daily target
- **THEN** the AI feedback is supportive, acknowledges the gap, and suggests a recovery plan
